# Краткая пояснительная записка

## 1. Цель и постановка задачи

Цель проекта — разработать MVP веб-приложения для проведения квизов, в котором организатор может создавать и запускать квизы, а участники — подключаться по коду комнаты и проходить опрос в реальном времени.

В рамках проекта были поставлены следующие задачи:

- реализовать регистрацию и авторизацию пользователей с разделением ролей (`organizer`, `participant`);
- реализовать создание и настройку квиза (название, категория, правила, время на вопрос);
- реализовать добавление вопросов разных типов (текст/изображение, один/несколько правильных ответов);
- реализовать проведение квиза в реальном времени;
- реализовать подсчет баллов и формирование итогового лидерборда;
- реализовать личные кабинеты и историю участия/проведенных сессий;
- обеспечить хранение данных и базовую готовность к релизному использованию.

## 2. Обоснование выбора средств разработки

Для реализации проекта выбран стек JavaScript на всем пути данных — от интерфейса до сервера.

### 2.1 Клиентская часть: React + Vite

- **React** выбран как компонентный UI-фреймворк для построения сложного интерфейса с ролями и состояниями.
- **Vite** обеспечивает быстрый запуск и сборку, удобен для MVP и активных итераций.
- Структура интерфейса разбита на страницы и переиспользуемые компоненты.

### 2.2 Серверная часть: Node.js + Express

- **Node.js** позволяет использовать единый язык разработки (JavaScript) на frontend и backend.
- **Express** обеспечивает простой и понятный REST API, удобную маршрутизацию и middleware-подход.

### 2.3 Real-time взаимодействие: Socket.IO

- Для синхронного показа вопросов всем участникам выбран **Socket.IO**.
- Используется событийная модель (`room:join`, `question:show`, `leaderboard:update`, `quiz:finished`), что позволяет в реальном времени обновлять состояние квиза.

### 2.4 Хранение данных: SQLite

- Для MVP выбрана **SQLite** как легковесная БД без отдельного сервера.
- Подходит для быстрого прототипирования и локального/малого продакшн-сценария.

### 2.5 Безопасность и валидация

- Аутентификация реализована через **JWT**.
- Разграничение доступа — по ролям пользователей.
- Валидация входных данных реализована через **Zod**.

## 3. Основные этапы разработки

### 3.1 Проектирование модели данных

Определены основные сущности: `users`, `quizzes`, `questions`, `question_options`, `quiz_sessions`, `session_participants`, `answers`.  
Заданы ключевые связи и ограничения целостности (уникальность email, room code, связки session/user/question и др.).

Результат этапа:

- сформирована устойчивая модель хранения;
- обеспечена возможность подсчета баллов, истории и лидерборда.

### 3.2 Разработка клиентской части

Реализованы:

- страницы авторизации и регистрации;
- кабинеты участника и организатора;
- интерфейс создания, редактирования и удаления квизов;
- интерфейс управления вопросами (добавление/редактирование/удаление);
- страница live-сессии с разными режимами для участника и организатора;
- отображение итогового лидерборда и истории.

Отдельно доработаны UX-элементы: уведомления, модальные окна, визуальные состояния кнопок, адаптивность.

### 3.3 Разработка серверной части

Сервер реализован как модульное приложение:

- `routes` (auth/quizzes/sessions/history/uploads/health);
- `middleware` (auth/upload);
- `runtime` (логика активных сессий, таймеры, завершение вопросов/квиза);
- `socket` (обработка real-time событий).

Реализованы API-операции для полного жизненного цикла квиза и сессии.

### 3.4 Настройка real-time логики

Ключевая логика проведения:

- участники подключаются в комнату по коду;
- вопрос активен только в ограниченный интервал времени;
- ответ принимается один раз на вопрос;
- после каждого шага обновляется состояние лидерборда;
- финальный лидерборд фиксируется после завершения квиза.

Дополнительно реализован tie-break: при равных баллах выше располагается участник с более быстрыми корректными ответами.

### 3.5 Тестирование

Проведено функциональное тестирование ключевых сценариев:

- регистрация и авторизация для двух ролей;
- создание и редактирование квизов/вопросов;
- запуск сессии и подключение по коду;
- ограничения времени ответа;
- корректность подсчета баллов и финального лидерборда;
- корректность отображения истории.

### 3.6 Развертывание

Проект подготовлен к развертыванию в контейнере:

- настроен `Dockerfile` для fullstack-сборки;
- настроен деплой на Fly.io;
- добавлены параметры окружения и постоянное хранилище для SQLite/загрузок.

## 4. Итоги реализации

В результате разработан работоспособный MVP, который позволяет:

- организатору создавать, настраивать и проводить квизы;
- участникам подключаться к активной сессии и отвечать в реальном времени;
- автоматически определять победителя по итоговым баллам (с учетом скорости при равенстве);
- сохранять данные о сессиях и показывать историю пользователям.

Реализованная версия соответствует ключевым функциональным требованиям и может использоваться как база для дальнейшего развития.

## 5. Перспективы развития

Планируемые улучшения:

- переход на PostgreSQL для масштабирования;
- расширенная аналитика по сессиям и вопросам;
- более гибкая система ролей и прав;
- CI/CD и автоматизированные интеграционные тесты;
- расширение форматов вопросов и режимов игры.
